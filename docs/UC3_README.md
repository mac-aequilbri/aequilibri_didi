# UC3 — MSME Coordinator (Multi-tenant Construction Project Management)

Developer reference for the UC3 module of the `aequilibri-next` application.

---

## What It Does

UC3 is a multi-tenant construction project management platform serving small-to-medium
enterprises (SMEs). Each tenant (contractor, builder, or PM firm) has isolated data and manages
one or more construction projects.

Key capabilities:

| Area | Description |
|---|---|
| Multi-tenancy | Tenant selected via cookie; all Prisma queries filter by `tenantId`. |
| Projects | CRUD for projects with status, client, start/end dates, and a `healthScore` (0–100). |
| Phases | Ordered project phases with completion tracking. |
| Action Items | Task tracking (open, in_progress, overdue, complete, cancelled). Can be created manually or auto-generated by the AI from meeting minutes. |
| Risks | Risk register with likelihood × impact scoring. Auto-escalation for scores ≥ 15. |
| Budget | Line items per project/phase with estimated vs actual tracking. Actuals are **blocked from AI write**. |
| Cashflow | Monthly projected vs actual entries per project. |
| Variation Orders (VOs) | Scope-change requests with cost/time impact. Can be manually created or AI-drafted. Require approval. |
| Documents | Text-content documents with AI contract analysis (key clauses, obligations, risk flags). |
| Meeting Minutes | Paste raw minutes → AI extracts action items as JSON → user confirms → actions created in bulk. |
| Weekly Reports | AI-generated markdown reports (executive summary, phases, budget, risks, actions). Draft → approved → sent lifecycle. |
| AI Chat | Conversational assistant scoped to a tenant/project with an explicit CRUD authority policy enforced via the system prompt. |
| Client Portal | Token-based read-only project view for clients. Tokens have optional expiry and a view counter. |
| Accounting | Simulated connect/sync for Xero, MYOB, and QuickBooks Online. No real OAuth is implemented. |
| Exec Log | Append-only log of every AI invocation with `aiAuthority` level and payload. |
| Delay Cascade | AI analysis of how a given delay propagates through phases and its cost/schedule implications. |

## How to Run

### Prerequisites

- Node.js 20+
- PostgreSQL **or** SQLite dev database

### 1. Install dependencies

```bash
npm install
```

### 2. Configure environment variables

```
DATABASE_URL=         # PostgreSQL connection string (production)
                      # Omit to fall back to SQLite dev.db
ANTHROPIC_API_KEY=    # Claude API key — if absent, AI features run in demo mode
```

### 3. Apply the database schema

```bash
# SQLite (local dev):
npm run db:dev:gen

# PostgreSQL (production):
npm run db:push
npm run db:generate
```

### 4. Start the dev server

```bash
npm run dev
```

Open [http://localhost:3000/uc3](http://localhost:3000/uc3).

If no tenant is selected the app redirects to `/uc3/select-tenant`. Select or create a tenant
first.

### Demo mode (no API key)

All `callClaude` calls return the canned demo response from `src/lib/claude.ts`. AI-generated
content (reports, VO drafts, document analysis, meeting extraction) will contain demo text and
the `ExecutionLog` rows will have `status = "demo"`.

## Request Flow

### Tenant resolution

Every Server Action and most server-component pages call `getTenantId()` (actions.ts:10–24)
or `getActiveTenant()` (`src/lib/uc3-tenant.ts`):

```
1. Read cookie uc3_tenant_id
2. If absent → query Uc3Tenant.findFirst({ where: { isActive: true } }) as fallback
3. If still null → redirect to /uc3/select-tenant
```

### Dashboard render

```
Browser GET /uc3
  → getActiveTenant() resolves tenantId
  → loadDashboard(tenantId) — four parallel Prisma queries:
      Uc3Project.findMany (last 5, select fields only)
      Uc3ActionItem.count (open)
      Uc3ActionItem.count (overdue)
      Uc3Risk.findMany (open, likelihood+impact only)
  → Uc3Project.count + groupBy status
  → highRisks = risks where likelihood × impact ≥ 15
  → Renders metrics grid + recent projects table
```

### AI Chat flow

```
User sends message
  → sendChatMessage Server Action (actions.ts:962):
      1. getTenantId() — redirect if absent
      2. Persist user Uc3ChatMessage
      3. Parallel context load:
            last 10 messages (tenant+project scoped)
            project details (if projectId provided)
            open action count
            open risk count
      4. Build system prompt with explicit CRUD policy:
            Full write:        ACTION_ITEMS, RISKS, VENDORS.rating, CASHFLOWS.projected
            Requires approval: PHASES, DECISIONS
            BLOCKED:           BUDGET.actual, PROJECT.creation, DOCUMENTS
      5. callClaude(systemPrompt, userMessage, {maxTokens: 1024})
      6. requiresApproval = response.includes("[REQUIRES_APPROVAL]")
      7. Persist assistant Uc3ChatMessage (requiresApproval flag)
      8. Write to Uc3ExecutionLog:
            aiAuthority = requiresApproval ? "approval_required" : "full_write"
      9. revalidatePath

User approves message
  → approveMessage Server Action:
      Uc3ChatMessage.update({ requiresApproval: false, approved: true })
```

Note: approving a message marks it as accepted. As with UC2, there is no code path that
automatically executes the described database write. ← ASSUMPTION based on code review.

### Meeting minutes → action items flow

```
User pastes raw minutes at /uc3/meeting-minutes/new
  → processMeetingMinutes Server Action:
      1. Create Uc3MeetingMinutes with status = "raw"
      2. callClaude with system prompt:
            "Return ONLY a JSON array: [{action,owner,due_date,priority}]"
      3. Strip markdown fences, JSON.parse the response
      4. Update Uc3MeetingMinutes: extractedActions, actionsCount, status = "processed"
      5. Write Uc3ExecutionLog with aiAuthority = "full_write"
      6. Redirect to /uc3/meeting-minutes/[id]

User clicks Confirm on the extracted actions page
  → confirmMeetingMinutes Server Action:
      1. Parse extractedActions JSON from the minutes row
      2. db.$transaction — create one Uc3ActionItem per entry (createdByAi = true)
      3. Mark minutes status = "confirmed"
      4. Redirect to /uc3/actions
```

### Weekly report generation flow

```
User selects project + week-ending at /uc3/reports/generate
  → generateWeeklyReport Server Action:
      1. Parallel DB load: project, phases, open actions (≤20), open risks (≤10), budgets
      2. Serialize context as JSON → callClaude(systemPrompt, contextStr, {maxTokens:1500})
         System prompt: generate markdown with executive summary, phases, budget, risks,
                        actions, recommendations
      3. Create Uc3WeeklyReport (status = "draft", isAiGenerated = true)
      4. Write Uc3ExecutionLog with aiAuthority = "approval_required"
      5. Redirect to /uc3/reports/[id]

User approves report → status = "approved"
User marks sent   → status = "sent"
```

### Variation order AI draft flow

```
User enters scope-change description at /uc3/variations/new (AI Draft tab)
  → aiDraftVariation Server Action:
      1. callClaude with system prompt:
            "Return ONLY a JSON object: {title,description,cost_estimate_rationale,risk_flags}"
      2. Strip markdown fences, JSON.parse
      3. Generate sequential refNumber (VO-{projectId}-{seq})
      4. Create Uc3VariationOrder (status = "draft", isAiDrafted = true, costImpact = 0)
      5. Write Uc3ExecutionLog (aiAuthority = "approval_required")
      6. User must manually edit cost/time impact and approve
```

### Document analysis flow

```
User submits document at /uc3/documents/new
  → createDocument stores text in Uc3Document.fileContent

User visits /uc3/documents/[id]/analyze → triggers analyzeDocument Server Action:
  1. callClaude — system prompt: "construction contract analyst, identify key clauses,
     financial terms, obligations, risk flags, recommended actions"
  2. User message: doc.fileContent.slice(0, 4000)   ← hard 4 000-char limit
  3. Update Uc3Document.aiAnalysis + analyzedAt
  4. Write Uc3ExecutionLog (aiAuthority = "blocked" — read-only AI action)
```

### Client portal flow

```
Contractor generates a token at /uc3/portal/[projectId]/manage
  → generatePortalToken: creates Uc3ClientPortalToken (randomBytes(32).toString("hex"))
    Optional: label, expiresAt

Client visits /uc3/portal/public/[token]
  → Public server component reads token, verifies isActive + optional expiry
  → Increments viewsCount
  → Renders read-only project summary  ← ASSUMPTION: exact content not verified; page file exists
```

## ML / AI Flow

### Model

`claude-opus-4-7` (hardcoded in `src/lib/claude.ts:7`). All calls are single-turn,
non-streaming, via `callClaude()`.

### CRUD authority policy

The AI chat system prompt encodes a three-tier write policy (actions.ts:1003–1009):

| Authority level | What the AI may do |
|---|---|
| `full_write` | ACTION_ITEMS, RISKS, VENDORS.rating, CASHFLOWS.projected — AI proposes; no extra gate |
| `approval_required` | PHASES, DECISIONS — AI prefixes response with `[REQUIRES_APPROVAL]` |
| `blocked` | BUDGET.actual, PROJECT.creation, DOCUMENTS — AI must not touch these |

The `aiAuthority` field is stamped on every `Uc3ExecutionLog` row for auditability.

**Important caveat**: the policy is enforced only through the system prompt instruction.
There is no server-side enforcement — the AI could propose a blocked write and the system
would accept it if the user confirms. ← ASSUMPTION: no guard code observed beyond the prompt.

### AI invocations summary

| Trigger | Input | Output | Max tokens |
|---|---|---|---|
| AI Chat | last 10 messages + context summary | Conversational response | 1 024 |
| Document analysis | `fileContent.slice(0, 4000)` | Clause/obligation analysis | 1 500 |
| Meeting minutes extraction | Raw minutes text | JSON array of action items | 1 500 |
| Weekly report generation | JSON of project, phases, actions, risks, budgets | Markdown report | 1 500 |
| VO AI draft | Scope-change description | JSON `{title, description, cost_estimate_rationale, risk_flags}` | 1 024 |
| Delay cascade analysis | JSON of project, phases, open risks + trigger/days | Narrative analysis | 1 500 |

## Manual Checks / Operator Runbook

### First-run: creating a tenant

1. Visit `/uc3/select-tenant`.
2. If no tenants exist in `Uc3Tenant`, create one via the UI or directly in the database.
3. Select the tenant — the `uc3_tenant_id` cookie is set and you land on the dashboard.

### Verifying tenant isolation

All Server Actions call `getTenantId()` and pass the result to every Prisma query as a
`where: { tenantId }` filter. To verify:
- Create two tenants with separate data.
- Switch between them via `/uc3/select-tenant`.
- Confirm each tenant sees only its own projects, actions, and risks.

### Checking the execution log

Every AI invocation writes to `Uc3ExecutionLog`. Review at `/uc3/exec-log`. Key fields:

| Field | What to check |
|---|---|
| `status` | `success` / `demo` / `error` |
| `aiAuthority` | `full_write` / `approval_required` / `blocked` |
| `result` | First 500 chars of Claude's response |
| `durationMs` | Not populated on all UC3 logs ← ASSUMPTION: field may be absent |

### Risk escalation

`escalateAllRisks` escalates all unescalated open risks with `likelihood × impact ≥ 15`
in a single `db.$transaction`. Visit `/uc3/risk-escalation` to trigger it. Verify by
checking `escalatedAt` and `escalationNote` on the affected rows.

### Client portal token lifecycle

1. Generate token at `/uc3/portal/[projectId]/manage`.
2. Share the URL `/uc3/portal/public/[token]` with the client.
3. Token can be deactivated via `deactivateToken` Server Action (sets `isActive = false`).
4. Expired tokens: expiry is stored in `expiresAt`; enforcement is handled by the public
   page — verify the page checks this field.  ← ASSUMPTION: page logic not fully read.

### Accounting integration

The accounting connection is fully simulated:
- `connectAccounting` stores `demo-token-${Date.now()}` as `accessToken` in plain text.
- `syncAccounting` increments `recordsSynced` by 3 and appends a log line.
- No real tokens are exchanged; no real data is fetched.
- Do not use this for production financial reconciliation without replacing with real OAuth flows.

## Known Risks & Limitations

Key: ✅ Fixed | ⚠ Open — mitigated | 🔴 Open — no fix yet

| Status | Risk | Detail |
|---|---|---|
| ✅ **Fixed** | **Silent tenant-1 fallback** | Both `getActiveTenant()` and the private `getTenantId()` in `actions.ts` fell back to `findFirst({ isActive: true })` when no cookie was present. Any request without a cookie silently operated as tenant 1. Fixed: both now return `null` immediately; all callers redirect to `/uc3/select-tenant`. |
| ✅ **Fixed** | **AI-drafted VOs always created with `costImpact = 0`** | The AI prompt returned only text fields; numeric cost and time fields were hardcoded to zero. Fixed: prompt now requests `cost_impact_dollars` (number) and `time_impact_days` (number); both are coerced and written to the VO row. |
| ✅ **Fixed** | **`confirmMeetingMinutes` bulk creation invisible in exec log** | The `$transaction` that creates action items from extracted minutes wrote no `Uc3ExecutionLog` entry. Fixed: a log row (`toolName: "minutes_confirm"`, `aiAuthority: "approval_required"`) is now written after the transaction. |
| ✅ **Fixed** | **Weekly report missing variation orders and cashflow** | `generateWeeklyReport` fetched project, phases, actions, risks, and budgets only. Open VOs and cashflow actuals were omitted, so the report could describe a project as on-budget while approved VOs existed. Fixed: both `uc3Cashflow` and open `uc3VariationOrder` are now fetched and included in the AI context. |
| ✅ **Fixed** | **`aiAuthority: "blocked"` mislabelled human writes** | `updateBudgetLine` and `updateCashflowActual` logged `aiAuthority: "blocked"`, implying the AI attempted the write. These are user-initiated actions. Fixed: changed to `"human_action"` on both. |
| ✅ **Fixed** | **`uc3_tenant_id` cookie missing `Secure` and `SameSite`** | Cookie was set with `sameSite: "lax"` and no `Secure` flag. Fixed: `setActiveTenant` now sets `sameSite: "strict"` and `secure: process.env.NODE_ENV === "production"`. |
| ✅ **Fixed** | **Document analysis truncation silent** | `analyzeDocument` silently passed only the first 4 000 characters to Claude with no user-visible warning. Fixed: the analyse page now shows an amber warning with the actual character count and estimated line cutoff when `fileContent.length > 4000`. |
| ✅ **Fixed** | **`analyze/page.tsx` retained stale tenant fallback** | The document analyse page had its own inline tenant-fallback logic copied before the D4 fix was applied. Fixed: replaced with a call to `getActiveTenant()`. |
| ⚠ **Mitigated** | **Accounting `accessToken` stored in plain text** | `connectAccounting` writes the token field unencrypted. Currently only `demo-token-*` values are stored, so no real credential is at risk. A `SECURITY TODO` comment has been added to the source. Before wiring a real OAuth flow, this column must use server-side encryption (e.g. AES-256-GCM with a key in an environment secret). |
| ⚠ **Mitigated** | **Meeting minutes JSON parse failure is silent** | If Claude returns malformed JSON, `processMeetingMinutes` stores `extractedActions = "[]"` and marks status `"processed"` with no user-visible error. Mitigated by the exec log entry (the raw Claude response is now stored in the log). Check `actionsCount = 0` rows in `Uc3MeetingMinutes` after processing as a signal of parse failures. |
| 🔴 **Open** | **AI write policy is instruction-only** | The CRUD authority tiers (`full_write` / `approval_required` / `blocked`) are enforced only through the system prompt. A compliance failure or prompt injection could result in the AI proposing blocked writes. No server-side guard rejects such proposals before they reach the user. Requires structured tool-use responses to enforce properly. |
| 🔴 **Open** | **Approving an AI message does not execute writes** | `approveMessage` sets `approved = true` on the chat message row only. No code path reads the approved message and performs the described database mutation. ← ASSUMPTION based on code review. |
| 🔴 **Open** | **No authentication on any route** | Tenant isolation relies entirely on a cookie integer. Any user who can reach the server and set `uc3_tenant_id` to a known value can read and write that tenant's data. Requires an auth framework before production use. |
| 🔴 **Open** | **No row-level auth beyond `tenantId`** | All `updateMany` calls filter by `{ id, tenantId }`. Any user with the correct tenant cookie and a known record ID can mutate any record in that tenant regardless of who created it. |
| 🔴 **Open** | **`healthScore` is never updated** | `Uc3Project.healthScore` is set to `100` on creation and no server action ever recalculates it. The value displayed on the dashboard and client portal is stale immediately after the first risk or overdue action is created. ← ASSUMPTION: no update logic found in `actions.ts`. |
| 🔴 **Open** | **No real-time updates** | All pages use `force-dynamic` rendering but there is no WebSocket or polling. Concurrent edits by two users will silently overwrite each other; the last write wins. |
